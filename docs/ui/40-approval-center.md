# 审批中心

> 日期：2026-03-27
> 用途：定义跨项目审批收件箱和审批详情页

## 1. 页面一：审批中心

### 1.1 页面卡片

- 页面名：`审批中心`
- 目标用户：审批人、项目负责人、管理员
- 页面目标：统一处理待审批请求，支持批准、拒绝、补充信息
- 页面入口：工作台待审批卡、项目阻塞卡
- 核心动作：
  - 浏览待审批项
  - 打开审批详情
  - 批准 / 拒绝 / 要求补充

### 1.2 信息结构

- 顶部：筛选器
- 左侧：审批列表
- 右侧：审批详情

### 1.3 桌面端线框

```text
+------------------------------------------------------------------------------------------------+
| 审批中心                                              [待处理] [全部] [按项目] [按风险]       |
+--------------------------------+----------------------------------------------------------------+
| 审批列表                       | 审批详情                                                       |
| CRM / Internal KB 访问         | Project：新一代 CRM 平台建设                                   |
| 招聘 / Offer 发放审批          | Team：软件开发 Team                                            |
| GitLab 写权限审批              | 触发阶段：权限与知识库接入                                     |
|                                | 请求内容：访问 internal-kb 行业资料                            |
|                                | 目的：生成客户行业知识辅助                                     |
|                                | 影响范围：报表、客户画像、行业分析                             |
|                                | 风险等级：high                                                 |
|                                | [批准] [拒绝] [要求补充]                                       |
+--------------------------------+----------------------------------------------------------------+
```

### 1.4 关键交互

1. 点击审批项查看请求上下文、风险和影响范围。
2. 选择 `要求补充` 时，填写补充问题并回流项目协作台。
3. 批准或拒绝后，列表即时更新，项目状态同步变更。
4. 请求发起人从项目里打开同一请求时，进入只读跟踪态，而不是审批态。

### 1.5 状态设计

- 空状态：展示“当前没有待处理审批”
- 加载状态：列表和详情骨架
- 成功状态：审批操作后给出审计确认
- 错误状态：审批提交失败允许重试
- 权限不足状态：只读用户可查看审批记录但不能操作

### 1.6 平台范围约束

- 当前版本只定义桌面端审批中心，不提供移动端独立方案
- 视口缩窄时优先压缩详情区与辅助信息，不把审批流重写成移动端全屏页

## 2. 页面二：审批详情抽屉

### 2.1 页面卡片

- 页面名：`审批详情`
- 目标用户：审批人
- 页面目标：在不离开当前上下文的情况下完成审批
- 页面入口：审批列表、项目阻塞提示、artifact 引用
- 核心动作：
  - 理解请求
  - 查看来源 run 和 artifact
  - 作出审批决定

### 2.2 桌面端线框

```text
+--------------------------------------------------------------+
| 审批详情                                                  X |
|--------------------------------------------------------------|
| 请求：Internal KB 访问                                      |
| 来源项目：新一代 CRM 平台建设                                |
| 来源任务：行业知识增强                                       |
|                                                              |
| 请求原因                                                     |
| 为补充客户行业资料，需要读取企业内部知识库的行业分析文档      |
|                                                              |
| 相关 artifact                                                |
| - 风险清单 v0.2                                              |
| - 行业字段映射草案                                           |
|                                                              |
| 影响评估                                                     |
| 批准后将恢复被阻塞的报表和客户画像工作流                      |
|                                                              |
| 决策说明 [..............................................]    |
|                                         [拒绝] [批准]        |
+--------------------------------------------------------------+
```

### 2.3 关键交互

1. 从项目中打开时，抽屉覆盖当前页，不破坏原上下文。
2. 审批说明强制要求填写，便于进入 `AuditLog`。

### 2.4 状态设计

- 空状态：无
- 加载状态：抽屉内部骨架
- 成功状态：显示“已写入审计日志”
- 错误状态：提交失败不关闭抽屉
- 权限不足状态：主按钮禁用并显示原因

## 3. 页面三：请求人审批跟踪抽屉

### 3.1 页面卡片

- 页面名：`审批跟踪`
- 目标用户：审批请求发起人、项目成员
- 页面目标：让无审批权成员在项目内查看审批流向和当前状态
- 页面入口：项目总览待审批卡、阻塞卡、协作台审批插条
- 核心动作：
  - 查看审批提交状态
  - 查看当前审批人
  - 确认已提交审批
  - 查看补充材料请求

### 3.2 桌面端线框

```text
+--------------------------------------------------------------+
| 审批跟踪                                                  X |
|--------------------------------------------------------------|
| 请求：Internal KB 访问                                      |
| 当前状态：等待审批                                           |
| 审批流向：已提交给 企业高层 A                                |
|                                                              |
| 请求原因                                                     |
| 为补充客户行业资料，需要访问企业内部知识库行业分析文档        |
|                                                              |
| 当前说明                                                     |
| 你没有审批权限，可在此查看状态或补充材料                      |
|                                                              |
| 附加材料 [上传补充说明]                                      |
|                                                              |
|                                         [确认已提交]         |
+--------------------------------------------------------------+
```

### 3.3 关键交互

1. 该抽屉不显示 `批准 / 拒绝` 主操作。
2. 如果审批人要求补充，抽屉中出现补充问题，并允许把材料回流项目协作台。
3. 审批完成后，状态自动刷新为 `已批准` 或 `已拒绝`。
