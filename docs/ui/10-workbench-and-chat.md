# 工作台与轻量会话

> 日期：2026-04-03
> 用途：定义首页、轻量聊天入口和最近记录页

## 1. 页面一：工作台

### 1.1 页面卡片

- 页面名：`工作台`
- 目标用户：所有登录用户
- 页面目标：作为每个用户的权限化驾驶舱，提供首页总览、快速发起和待办聚合，同时明确支持临时 Chat 与正式 Project 两种工作模式
- 页面入口：侧边栏 `工作台`
- 核心动作：
  - 新建对话
  - 跳转最近项目
  - 查看待审批项
  - 查看运行中的项目

### 1.2 信息结构

- 左侧：全局导航
- 中上：欢迎区与主输入框
- 中下左：最近项目 / 最近会话
- 中下中：待审批 / 风险提示
- 中下右：运行中 Team / 今日计划
- 顶部可扩展：管理视角下出现组织级统计卡

### 1.3 桌面端线框

```text
+------------------------------------------------------------------------------------------------+
| 工作台                                        [我的范围 v] [全局搜索] [新建项目]              |
+------------------------------+-----------------------------------------------------------------+
| 侧边栏                       | 你好，Kr Li                                                      |
|                              | 今天你要推进什么？                                               |
|                              | +-------------------------------------------------------------+  |
|                              | | 输入一句目标，先开启临时对话或跳转到已有项目               |  |
|                              | +-------------------------------------------------------------+  |
|                              | [新建对话] [新建项目] [查看待审批]                              |
|                              |-----------------------------------------------------------------|
|                              | 最近项目               | 待审批                | 正在运行的 Team |
|                              | - 新一代 CRM 平台建设  | - KB 访问审批         | - 软件开发 Team |
|                              | - 招聘流程优化         | - GitLab 写权限审批   | - 招聘 Team     |
|                              |-----------------------------------------------------------------|
|                              | 最近会话               | 今日摘要                                 |
|                              | - 昨天的需求讨论       | 3 个项目推进中，1 个阻塞，2 个审批待处理 |
|                              | - GitLab 接入方案      |                                           |
+------------------------------+-----------------------------------------------------------------+
```

### 1.3.1 管理视角扩展示意

同一页面在高权限下增加组织级卡片，而不是跳到另一套页面：

```text
+------------------------------------------------------------------------------------------------+
| 工作台                                        [全组织 v] [全局搜索] [新建项目]                |
+------------------------------+-----------------------------------------------------------------+
| 侧边栏                       | 组织级摘要：12 个项目进行中 / 3 个阻塞 / 5 个待审批             |
|                              |-----------------------------------------------------------------|
|                              | 项目健康表             | 风险热区              | Team 负载          |
|                              | CRM 平台建设  medium   | 招聘流程优化  high   | 软件开发 Team  6   |
|                              | 销售培训体系  low      | SSO 依赖       medium | 招聘 Team      4   |
+------------------------------+-----------------------------------------------------------------+
```

### 1.4 关键交互

1. 在主输入框输入目标，系统先判断是否进入 `新建对话`。
2. 点击 `新建项目`，进入项目创建页。
3. 点击待审批卡片，跳转 `审批中心` 并自动过滤到“与我相关”。
4. 点击最近项目，直接进入项目总览。
5. 高权限用户可切换 `我的范围 / 部门范围 / 全组织` 数据视角。

补充原则：

- 如果用户只是想快速问问题、试一个想法或做短任务，默认先进入 `新建对话`
- 如果用户已经明确要共享、治理、审批或持续推进，才优先进入 `新建项目`

### 1.5 状态设计

- 空状态：展示引导卡，说明 `新建对话` 与 `新建项目` 的差别
- 加载状态：最近列表和摘要卡使用骨架屏
- 成功状态：显示今日项目健康度摘要
- 错误状态：某个聚合模块加载失败时仅局部报错
- 权限不足状态：没有审批权限时隐藏审批汇总，只保留只读统计

### 1.6 权限分层规则

- 普通成员：
  - 看到与自己相关的项目、最近会话、待我处理审批
- 项目负责人：
  - 额外看到所负责项目的健康摘要、阻塞和风险
- 管理者 / 高层：
  - 额外看到跨项目统计、风险热区、Team 负载和审批堆积

因此，`工作台` 是统一页面，不是多个角色各做一套页面。

### 1.7 工作台卡片规则

工作台只保留五类卡片：

- 发起类：快速输入框、新建对话、新建项目、常用模板
- 执行类：最近项目、运行中的 Team、活跃 Workstream、今日进展
- 治理类：待我审批、审批堆积、阻塞项目、风险热区
- 资产类：最近 artifact、最近引用知识、定时任务结果
- 组织类：项目健康表、Team 负载、本周里程碑、项目成员参与分布

默认优先级：

1. 治理类
2. 执行类
3. 发起类
4. 资产类
5. 组织类

显示规则：

- 无数据优先隐藏；关键入口保留入口态，不保留空列表态
- 权限先于个性化；用户可调顺序和折叠，但不能看到越权数据
- 范围切换只改数据集，不改页面结构

角色默认首屏：

- 普通成员：快速发起、最近项目、最近会话、我的待处理、活跃 Workstream、最近 artifact、今日进展
- 项目负责人：快速发起、项目健康度、阻塞与风险、待我审批、运行中的 Team、最近 artifact、今日进展
- 管理者：组织级摘要、项目健康表、风险热区、审批堆积、Team 负载、项目成员参与分布、本周完成里程碑

| 卡片名 | 类型 | 最低显示角色 | 点击后去向 |

### 1.9 成员参与卡片

为控制范围，`项目成员参与分布` 不做新页面，先作为管理视角下的一张工作台卡片。

推荐内容：

- 人类成员姓名
- 当前参与项目数
- 主要参与的项目
- 当前角色：发起人 / 审批人 / 参与成员

点击后动作：

- 跳转到项目列表，并自动按该成员过滤
- 右侧摘要区显示该成员参与的项目集合

这样可以满足“谁参与了哪些项目”的巡视需求，但不额外长出新的分析模块。

### 1.10 平台范围约束

- 当前版本只做桌面 Web，不做移动端产品方案
- 工作台按桌面端信息密度设计，默认以宽屏与笔记本视口为基线
- 窄窗口仅做浏览器缩放容错，不新增移动端专属信息架构

## 2. 页面二：新建对话 / ChatSession

### 2.1 页面卡片

- 页面名：`新建对话`
- 目标用户：所有登录用户
- 页面目标：承接探索性、个人化、低治理成本的初始需求输入；它本身就是完整产品能力，不是阉割版
- 页面入口：侧边栏 `+ 新建对话`、工作台主输入框
- 核心动作：
  - 发起探索性需求
  - 上传参考资料
  - 生成摘要
  - 提升为 Project

### 2.2 信息结构

- 左侧：最近会话列表
- 中间：对话区
- 右侧：摘要与提升卡

### 2.3 桌面端线框

```text
+------------------------------------------------------------------------------------------------+
| 新建对话                                                     [上传资料] [提升为 Project]      |
+------------------------------+--------------------------------+--------------------------------+
| 最近会话                     | 对话区                         | 摘要区                         |
| - CRM 需求探索               | 用户：我们想做一个 CRM MVP... | 目标：销售 CRM MVP             |
| - 招聘流程设计               | Agent：我先整理目标和约束...  | 范围：客户/联系人/商机...      |
|                              |                                | 约束：SSO / 3 个月 / Web 优先  |
|                              | [输入框......................................................] |
|                              |                                | [生成摘要] [提升为 Project]    |
+------------------------------+--------------------------------+--------------------------------+
```

### 2.4 关键交互

1. 用户输入目标，系统实时生成结构化摘要。
2. 用户上传材料后，右侧摘要卡追加“已引用资料”。
3. 点击 `提升为 Project`，进入新建项目页并预填目标、范围、约束。

补充运行规则：

- 进入 `新建对话` 时先处于前端 draft 状态，不立刻创建正式会话记录。
- 首次成功发送后，页面应原地切换到正式 `ChatSession`，不能因为真实 id 返回而整页 remount。
- 消息流、上传文件、摘要卡和右侧运行细节必须共享同一会话运行上下文。
- 内部 `thread` 可作为实现锚点存在，但不应在普通用户主界面里直接作为主标题或主对象名暴露。

### 2.5 状态设计

- 空状态：展示示例 prompt 和“何时该提升为项目”
- 加载状态：对话流式输出，摘要区显示提炼中
- 成功状态：摘要卡可直接复用
- 错误状态：对话失败可重试；摘要提取失败不影响继续聊天
- 权限不足状态：无项目创建权限时隐藏 `提升为 Project`

补充原则：

- `新建对话` 不应在 UI 上被呈现成“项目前置步骤”
- 只有当用户显式需要项目能力时，才把 `提升为 Project` 作为增强动作出现

### 2.5.1 右侧上下文区职责

右侧区域不是独立工具栏，而是当前 `ChatSession` 的统一上下文面板。

它应承接：

- 结构化摘要
- 已引用资料
- 运行状态摘要
- 可提升为 `Project` 的关键信息

它不应承接：

- 对普通用户暴露底层 `thread_id`
- 与当前会话无关的全局治理信息

### 2.6 平台范围约束

- `新建对话 / ChatSession` 当前只定义桌面端主工作面
- 设计重点放在三栏或主副栏桌面布局，不为移动端单独设计抽屉式结构
- 窄窗口只允许收缩次级区域，不改变“对话优先、上下文其次”的层级

## 3. 页面三：最近记录

### 3.1 页面卡片

- 页面名：`最近记录`
- 目标用户：所有登录用户
- 页面目标：集中浏览最近访问的对话、项目和 artifact
- 页面入口：侧边栏 `最近`
- 核心动作：
  - 回到最近会话
  - 回到最近项目
  - 快速清理无用记录

### 3.2 信息结构

- 顶部：筛选器
- 中部：按类型分组的最近记录列表
- 右侧：所选记录详情

### 3.3 桌面端线框

```text
+-----------------------------------------------------------------------------------------------+
| 最近记录                                              [全部] [对话] [项目] [Artifact]        |
+--------------------------------+----------------------------------------------------------------+
| 列表                           | 详情                                                           |
| 今天                           | 标题：昨天的需求讨论                                          |
| - 昨天的需求讨论               | 类型：ChatSession                                              |
| - 新一代 CRM 平台建设          | 最近动作：已提升为 Project                                    |
| 昨天                           | 关联：Project / 新一代 CRM 平台建设                           |
| - GitLab 接入方案              | [打开] [从最近中移除]                                         |
+--------------------------------+----------------------------------------------------------------+
```

### 3.4 关键交互

1. 用户按类型筛选最近项。
2. 点击记录后，右侧展示来源、最近动作、关联对象。
3. 允许“从最近中移除”，但不删除真实对象。

### 3.5 状态设计

- 空状态：提示用户从工作台发起第一条对话或进入第一个项目
- 加载状态：分组列表使用骨架行
- 成功状态：自动按时间分组
- 错误状态：仅影响最近模块，不影响其他页面
- 权限不足状态：若关联对象已无权限，只显示“你已无访问权限”

### 3.6 平台范围约束

- `最近记录` 当前只承诺桌面端列表 + 详情布局
- 若视口缩窄，优先压缩边栏与详情宽度，而不是重新发明移动端二级页流转
