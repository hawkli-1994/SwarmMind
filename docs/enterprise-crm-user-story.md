# 企业级 CRM 项目用户故事

> 日期：2026-03-27
> 用途：用端到端用户故事校验 SwarmMind 架构是否闭环
> 关联文档：`docs/architecture.md`、`docs/team-interface-agent-adr.md`、`docs/workflow-template-system.md`

## 1. 故事目标

本文通过一个企业员工发起 CRM 项目的完整故事，验证以下架构设计是否成立：

- `Project` 是否足以承担企业级协作工作空间
- `AgentTeam` 模板与 `ProjectTeamInstance` 实例是否清晰分离
- `TeamInterfaceAgent` 是否能作为多人协作入口
- Team 工作流模板是否能在 `Project` 内驱动可视化执行
- `RBAC` 与跨层级审批流是否能支撑真实企业治理
- 高层驾驶舱是否能建立在 `Project` / `Run` / `Audit` 之上

## 2. 角色

### 2.1 人类角色

- 企业员工用户：项目发起人，来自业务部门，负责提出需求、补充约束、反馈意见
- 企业高层 A：审批人之一，拥有更高权限，可审批跨部门资源与敏感权限
- 其他人类员工：可能参与评审、提供业务规则、验收结果

### 2.2 AI 角色

- `TeamInterfaceAgent`：软件开发 Team 在项目内的统一接口人
- 产品经理 Agent
- 架构师 Agent
- Tech Lead Agent
- 前端工程师 Agent
- 后端工程师 Agent
- QA Agent
- 交付协调 Agent

## 3. 场景概述

一家企业准备开发新的 CRM 系统。

业务部门员工登录 SwarmMind 平台后，创建一个新的 `Project`，命名为“新一代 CRM 平台建设”。
他在项目创建页填写：

- 项目目标：构建企业统一 CRM
- 业务范围：客户管理、销售漏斗、商机跟进、权限管理、报表
- 约束：优先支持 Web 端，三个月内交付 MVP，必须接入公司现有单点登录
- 可见范围：业务部门、技术部门、管理层可见

项目创建完成后，这个 `Project` 成为该事项的唯一协作边界。

## 4. 故事主线

### 4.1 创建 Project

企业员工用户点击“新建项目”，系统创建：

- `Project`
- 初始 `ProjectMember` / `RoleBinding`
- 空的 `TaskStore`
- 空的 `RunStore`
- 空的项目看板与概览页

在架构上，这一步验证：

- 所有执行都必须先落在 `Project` 内
- 项目可见性、权限和审计先于执行建立

### 4.2 选择 Team

用户在项目设置页选择一个可复用的 `AgentTeam` 模板：`软件开发 Team`。

这个模板不是直接开始干活，而是被绑定到该项目，系统创建：

- `ProjectTeamBinding`
- `ProjectTeamInstance`
- `TeamInterfaceAgent`

此时用户看到的是：

- 项目内已挂载“软件开发 Team”
- Team 当前状态：待需求澄清
- Team 入口：接口人 Agent 在线

在架构上，这一步验证：

- Team 是模板，不是执行实体
- 同一个 Team 模板可以被不同项目复用
- 进入项目后必须先实例化为 `ProjectTeamInstance`

### 4.3 与接口人 Agent 对话

企业员工用户打开项目聊天入口，与 `TeamInterfaceAgent` 对话：

“我们要做一个 CRM，先做销售团队可用的 MVP。希望支持客户录入、联系人管理、商机阶段推进、销售跟进记录、基础权限管理和销售报表。必须接公司 SSO，未来还要支持移动端，但第一期先不做。”

接口人 Agent 先做 intake：

- 识别项目目标与范围
- 提炼业务优先级
- 标记关键约束
- 识别这是一个新的 `Workstream`
- 创建工作流上下文

然后它给出整理后的确认摘要：

- 第一阶段目标：销售 CRM MVP
- 必要模块：客户、联系人、商机、跟进记录、权限、报表
- 非目标：移动端、复杂自动化营销
- 强约束：公司 SSO、MVP 三个月

用户确认后，Team 正式进入执行。

在架构上，这一步验证：

- `TeamInterfaceAgent` 不是简单聊天机器人，而是工作入口治理者
- `Workstream` 可以把“新项目启动”识别为一条新的持续工作流
- 项目内 thread 和后续执行都从入口人开始归档

### 4.4 工作流模板启动

`软件开发 Team` 自带一套工作流模板，例如：

1. 需求澄清
2. 范围切分
3. 产品方案
4. 架构设计
5. 迭代规划
6. 开发执行
7. 测试验收
8. 上线准备

当接口人 Agent 确认需求后，系统基于工作流模板，在 `Project` 内自动生成：

- Epic / Feature / Task 层级任务
- 项目看板（Kanban）
- 项目 Overview
- 风险与阻塞区

如果模板认为有必要，也可以动态生成：

- 时间线与甘特图
- 系统模块关系图
- 依赖图
- 里程碑燃尽图
- 风险热力图

这些都是 `Project` 的可视化控制面视图，而不是 Team 私有界面。

用户此时能看到：

- 需求阶段正在进行
- 下一个阶段是什么
- 当前有哪些 Agent 在活跃工作
- 哪些任务已完成、进行中、阻塞中

在架构上，这一步验证：

- Team 工作流模板可以驱动项目内结构化执行
- 可视化视图应建立在 `TaskStore`、`RunStore`、`ArtifactStore` 与 `AuditLog` 上
- `Kanban` 是最低必须支持的 `Project` 视图，其他图表按模板能力动态启用

### 4.5 多 Agent 协作展开

随后，各类 Agent 在 `ProjectTeamInstance` 内轮番上阵。

典型过程如下：

#### 产品经理 Agent

- 输出 PRD 草案
- 拆分 MVP 范围
- 识别关键业务流程
- 在看板中生成产品任务卡片

#### 架构师 Agent

- 设计系统边界
- 识别 CRM 模块
- 产出初步架构图
- 标注 SSO 接入点和权限域

#### Tech Lead Agent

- 把方案拆成开发迭代
- 划分前后端工作包
- 识别关键风险与依赖

#### 前端 / 后端工程师 Agent

- 分别推进 UI、接口、数据模型、权限逻辑等任务
- 在任务卡片上持续更新状态
- 产出设计稿、接口说明、代码 patch、测试结果等 artifact

#### QA Agent

- 编写测试方案
- 根据 acceptance criteria 回归验证
- 标记缺陷与待修复项

在用户界面上，企业员工用户不是只看到最终答案，而是持续看到：

- 看板列的变化
- 甘特图时间推进
- 任务之间的 handoff
- Agent 讨论与分解摘要
- 关键 artifact 的逐步形成

在架构上，这一步验证：

- `RunStore` 必须能承接持续推进过程，而不是只保存一次性回答
- `ArtifactStore` 必须承接 PRD、架构图、测试报告、代码交付物等
- `AuditLog` 必须能记录重要决策，而不是只记录最终结果

### 4.6 用户中途干预与指导

项目推进过程中，企业员工用户并不是旁观者。
他可以随时进入项目，对正在推进的工作提出指导意见，例如：

- “MVP 先不要做复杂报表，只保留销售漏斗汇总”
- “客户字段里必须包含行业和地区”
- “我们需要兼容历史 CRM 的导入”
- “这个流程需要先给销售总监看一下”

这些指导会先进入 `TeamInterfaceAgent`，由它决定：

- 是否进入已有 `Workstream`
- 是否应复用当前 thread
- 是否要生成新任务
- 是否需要通知某个专业角色 Agent

在界面上，这些人类意见会反映为：

- 新增任务
- 调整优先级
- 修改里程碑
- 变更范围说明
- 新的讨论摘要

在架构上，这一步验证：

- 人类可以持续介入，而不是只能在开头提需求、结尾看结果
- 入口治理和 `thread_policy=reuse` 是必要的，否则多人指导会把上下文打乱

### 4.7 重大决策与权限审批暂停

推进到“权限与知识库接入”阶段时，架构师 Agent 识别到：

- CRM 需要读取企业内部知识库中的客户行业资料
- 当前项目成员和发起员工都没有该知识库的访问权限

此时 Team 工作流不会偷偷绕过去，而是进入等待审批状态。

系统自动生成一个审批请求，包含：

- 需要访问的知识库范围
- 访问目的
- 预计影响的功能模块
- 风险等级
- 请求发起上下文

企业员工用户在项目界面中看见：

- 当前阻塞点：知识库权限未获批准
- 当前状态：等待审批
- 审批流向：已提交企业高层 A

但该员工本人没有审批权，所以他只能点击“确认提交审批”，不能自己批准。

审批流随后自动流转到企业高层 A 的界面。

在架构上，这一步验证：

- `RBAC` 不只是看不看得见，还必须决定“谁可以批准什么”
- 审批对象是明确的 run / capability 请求，而不是模糊的人机对话
- 工作流可以暂停，并保持当前项目状态对所有相关方可见

### 4.8 企业高层 A 的审批视角

企业高层 A 登录后，在自己的工作台上收到一条审批通知。

他能看到：

- 这是哪个 `Project` 发起的请求
- 哪个 Team、哪个工作流阶段触发的
- 请求访问什么知识库
- 为什么需要该权限
- 如果批准，影响哪些任务推进

高层 A 审批通过后：

- 审批结果进入 `AuditLog`
- 能力或访问令牌按企业权限流程下发
- 项目从“等待审批”回到“继续推进”
- Team 工作流自动恢复

在架构上，这一步验证：

- 审批流可以跨人、跨层级流转
- 审批人和执行人可以完全不是同一批人
- `Project` 是承载审批上下文的核心边界

### 4.9 高层驾驶舱

企业高层 A 除了处理单次审批外，还有一个管理驾驶舱界面。

在这个界面上，他可以看到：

- 不同部门正在推进的 `Project`
- 每个项目当前阶段
- 哪些项目阻塞中
- 哪些项目风险偏高
- 哪些人类员工参与了哪些项目
- 哪些审批正在等待
- 哪些 Team 当前最繁忙

如果他点进某个项目，还能看到：

- 项目 Overview
- 看板和甘特图
- 关键 artifact
- 重要审批与决策记录
- 当前参与的人类成员与 AI Team

在架构上，这一步验证：

- 高层驾驶舱不是额外系统，而是 `ProjectStore`、`ProjectMemberStore`、`RunStore`、`AuditLog` 的聚合视图
- 人类成员参与度必须是正式数据，而不是临时 UI 状态

### 4.10 项目完成

随着产品、架构、开发、测试和验收逐步完成，系统中各项任务被推进到完成状态。

最终，用户在项目里看到：

- 看板全部收敛
- 甘特图显示主要阶段完成
- CRM MVP 交付完成
- 所有关键 artifact 已归档
- 审批链路完整可回溯
- 复盘文档和经验总结被沉淀

此时：

- 项目产物留在 `Project`
- Team 经验可被提炼后进入 `TeamAssetStore`
- 但不会自动写入 DeerFlow memory 形成第二真相源

在架构上，这一步验证：

- `Project` 是执行与归档边界
- Team 资产与项目产物必须分层
- 结束后仍应保留完整回放能力

## 5. 这个故事验证了什么

### 5.1 被验证的设计

- `Project` 作为企业级协作边界是必要且合理的
- Team 模板和项目内 Team 实例分离是必要的
- `TeamInterfaceAgent` 是多人协作下的重要入口治理层
- `RBAC` 是审批流和可见性控制的基础
- 看板、甘特图、风险视图等应属于控制面，不应塞进 DeerFlow 内部
- 高层驾驶舱可以建立在项目、成员、运行、审计数据之上

### 5.2 暴露出的进一步设计需求

这个故事也说明，后续还需要继续细化：

- 工作流模板如何定义和版本化
- Kanban / 甘特图 / 风险图的数据模型
- 审批对象的统一 schema
- capability 请求和企业权限系统如何映射
- 高层驾驶舱的聚合指标定义

## 6. 对当前架构的结论

如果 SwarmMind 要服务企业级项目协作，而不是单次问答，这条故事说明当前架构方向基本正确：

- `Project` 必须是顶层边界
- `ProjectTeamInstance` 必须是 Team 的真实运行承载体
- `TeamInterfaceAgent` 必须作为多人协作入口
- `RunStore`、`ArtifactStore`、`AuditLog` 必须是一等控制面数据
- `RBAC` 和跨层级审批流必须是正式架构组成

如果这些点缺失，用户看到的就不会是“项目在推进”，而只会是“一堆 Agent 回复了很多话”。
