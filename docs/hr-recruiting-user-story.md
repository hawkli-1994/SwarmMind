# HR 招聘项目用户故事

> 日期：2026-03-27
> 用途：用招聘场景验证 `Project` 框架是否能承接权限敏感、审批密集、带排期要求的正式业务协作
> 关联文档：`docs/enterprise-crm-user-story.md`、`docs/marketing-campaign-user-story.md`、`docs/ui/02-page-map-and-flows.md`、`docs/ui/30-projects-and-project-space.md`、`docs/ui/40-approval-center.md`、`docs/ui/60-knowledge-library-schedules.md`

## 1. 故事目标

本文选择 HR 招聘场景，验证以下问题：

- `Project` 是否能承接一个正式招聘项目，而不只是研发或营销活动
- 当前固定的六个项目子页，是否足以覆盖招聘启动、筛选、面试、审批、发 offer 和收尾
- 敏感候选人资料、薪酬信息、审批链路能否在现有权限和审计框架下闭环
- 面试排期、周报提醒、反馈催办能否在现有“项目 + 定时任务”组合下成立
- 在不扩张页面范围的前提下，当前 `工作台 -> 项目 -> 治理 -> 资产` 结构是否还能走通

## 2. 角色

### 2.1 人类角色

- HRBP：项目发起人，负责发起招聘、推进流程、协调候选人沟通
- 用人经理：明确岗位需求，参与筛选与录用决策
- 面试官：提交面试反馈
- 薪酬审批人：审批超带宽 offer 或特殊薪酬方案
- 法务 / 合规负责人：审批特殊用工条款或跨地区用工风险

### 2.2 AI 角色

- `TeamInterfaceAgent`：招聘 Team 在项目内的统一接口人
- 招聘策略 Agent
- JD 优化 Agent
- 初筛与候选人整理 Agent
- 面试协调 Agent
- Offer 方案 Agent
- 招聘分析 Agent

## 3. 场景概述

一家企业准备招聘一位“高级产品经理”，需要在 8 周内完成到岗。

HRBP 登录 SwarmMind 后，创建一个新的 `Project`，命名为“高级产品经理招聘”。
他在创建页填写：

- 项目目标：8 周内完成高级产品经理录用并到岗确认
- 业务范围：岗位澄清、JD、渠道投放、简历初筛、面试安排、反馈汇总、offer 发放
- 约束：HC 已批准、预算带宽固定、候选人资料仅限授权成员查看、关键面试反馈需在 24 小时内提交
- 可见范围：HR、用人部门负责人、管理层中与该岗位相关的成员

项目创建后，成为该岗位招聘推进、候选人协作、审批与归档的唯一正式边界。

## 4. 故事主线

### 4.1 创建 Project

HRBP 点击“新建项目”后，系统创建：

- `Project`
- 初始 `ProjectMember` / `RoleBinding`
- 空的 `TaskStore`
- 空的 `RunStore`
- 空的项目总览、看板、产物空间

这里验证的是：

- 招聘也必须先进入正式 `Project`
- 岗位可见性、候选人资料权限和审计先于执行建立

### 4.2 选择 Team

HRBP 在创建页选择 `招聘 Team`，或者先建立空项目，稍后在设置页挂载。

挂载后系统创建：

- `ProjectTeamBinding`
- `ProjectTeamInstance`
- `TeamInterfaceAgent`

此时项目里能看到：

- 已挂载“招聘 Team”
- 当前状态：待岗位澄清
- 接口人 Agent 在线

这里验证的是：

- `Team` 依然是模板，进入项目后才变成真实运行实例
- 同一个招聘 Team 模板可以复用于多个岗位招聘项目
- 先建项目、后挂 Team 的路径在招聘场景下依然成立

### 4.3 与接口人 Agent 对话

HRBP 打开项目协作台，对 `TeamInterfaceAgent` 说：

“我们要在 8 周内招到一位高级产品经理，优先考虑有企业 SaaS 经验、做过销售增长或 CRM 方向的人。预算带宽固定在 P7 上限内。前两轮由用人团队完成，最终轮需要 VP 参与。候选人资料不能被无关成员看到，面试反馈请在 24 小时内收齐。”

接口人 Agent 做 intake：

- 识别岗位目标、时间窗口、预算和审批边界
- 提炼 must-have 与 nice-to-have
- 标记候选人资料和薪酬信息是敏感内容
- 建立“高级产品经理招聘启动” `Workstream`

然后给出确认摘要：

- 第一阶段目标：确认岗位画像并启动招聘
- 必要交付：JD、筛选标准、面试流程、反馈模板、offer 方案
- 强约束：预算带宽、权限隔离、反馈时效、关键轮次审批
- 关键协作方：HR、用人经理、面试官、薪酬审批人

用户确认后，Team 正式进入执行。

这里验证的是：

- `TeamInterfaceAgent` 同样适合招聘场景
- 招聘项目也需要统一入口治理，否则面试反馈、候选人沟通、审批会散掉

### 4.4 工作流模板启动

`招聘 Team` 自带一套默认工作流模板，例如：

1. 岗位澄清与 JD 确认
2. 渠道与候选人搜寻
3. 简历初筛与 shortlist
4. 面试安排与反馈回收
5. Offer 方案与审批
6. 候选人确认与入职前收尾
7. 项目复盘与资产沉淀

接口人 Agent 确认需求后，系统在项目内自动生成：

- 招聘任务骨架
- 项目看板
- 时间线与风险摘要
- 初始产物位

用户此时看到的是：

- 当前在哪个招聘阶段
- 哪些任务正在推进
- 哪些候选人相关事项被阻塞
- 哪些正式文档已经形成

这里验证的是：

- 固定 `Project` 骨架不依赖研发数据模型
- 招聘流程同样能投影到看板、时间线、产物和审批

### 4.5 多 Agent 协作展开

随后，招聘 Team 内的多个 Agent 进入协作：

#### 招聘策略 Agent

- 输出招聘策略 brief
- 建议渠道优先级
- 识别关键时间风险

#### JD 优化 Agent

- 产出 JD 草案
- 调整岗位要求表达
- 对齐用人经理反馈

#### 初筛与候选人整理 Agent

- 汇总候选人来源
- 整理 shortlist
- 标记高匹配与风险点

#### 面试协调 Agent

- 生成面试轮次安排建议
- 汇总待收反馈
- 识别超时未提交反馈的节点

#### Offer 方案 Agent

- 整理 offer 建议包
- 对比预算带宽与候选人期望
- 触发必要审批

#### 招聘分析 Agent

- 输出每周招聘进度摘要
- 分析渠道效果、流程转化和瓶颈

用户在项目里持续看到：

- 看板卡片变化
- 时间线推进
- 任务 handoff
- JD、面试模板、候选人摘要、offer 建议、招聘周报等 artifact 逐步形成

这里验证的是：

- `RunStore`、`ArtifactStore`、`AuditLog` 对招聘项目仍然是一等数据
- 用户看到的应是“招聘在推进”，不是“一堆零散消息”

### 4.6 用户中途干预

招聘推进到第三周，用人经理提出新要求：

- “行业经验比大厂背景更重要，放宽 title 要求。”
- “优先推进内推候选人，外部猎头渠道先降权。”
- “二面后必须补一轮 cross-team 面试，确认协作能力。”

这些意见进入 `TeamInterfaceAgent` 后，被转换为：

- JD 调整任务
- 渠道优先级调整
- 面试流程变更
- 时间线和风险更新

这里验证的是：

- 招聘项目同样需要持续校正，而不是一次性配置完毕
- `协作台` 能承接岗位口径和流程调整

### 4.7 审批暂停与恢复

推进到发 offer 阶段时，系统识别到两类审批：

- 候选人期望薪酬超出现有带宽，需要薪酬审批人确认
- 候选人涉及异地雇佣条款，需要法务 / 合规负责人确认

此时项目状态切换为 `等待审批`，系统生成明确审批请求，包含：

- 请求类型：薪酬带宽审批 / 特殊用工条款审批
- 关联项目与当前阶段
- 关联 artifact：offer 建议包、岗位预算说明、候选人摘要
- 风险等级与影响范围
- 若不批准，将阻塞哪些任务

HRBP 在项目总览里看到：

- 当前阻塞点：offer 薪酬待审批、特殊条款待确认
- 当前状态：等待审批
- 已提交给谁：薪酬审批人、法务负责人

但 HRBP 本人没有审批权，所以只能跟踪，不能自行批准。

审批通过后：

- 审批结果进入 `AuditLog`
- Offer 发放任务恢复执行
- 项目继续推进到候选人确认与收尾阶段

这里验证的是：

- 现有审批中心模型可以承接招聘中的关键治理节点
- 审批对象依然是明确请求，而不是聊天语义

### 4.8 排期、提醒与反馈催办

招聘推进过程中，HRBP 还需要处理大量节奏型事务：

- 安排候选人面试时间
- 提醒面试官提交反馈
- 每周给用人经理发送招聘进度摘要

在当前框架下，这些动作可以这样承接：

- 面试安排本身落在项目任务和时间线中
- 面试确认、反馈催办、周报发送可通过 `定时任务` 或项目级自动动作触发
- 最近执行结果、失败原因和重试入口在 `定时任务` 中回看

这里验证的是：

- 现有框架足以承接“节奏管理”
- 但一对一面试排期目前仍偏向“任务 + 外部日历集成”的表达，不是独立 ATS 排期模块

### 4.9 管理者在工作台巡视

招聘负责人登录后，在自己的 `工作台` 中看到：

- 进行中的岗位招聘项目
- 哪些项目存在审批阻塞
- 哪些岗位风险高，例如反馈超时、候选人流失风险上升
- 哪些成员参与多个招聘项目
- 招聘 Team 当前负载

点击“高级产品经理招聘”后，他进入项目总览，再下钻到看板、时间线、产物和审批跟踪。

这里验证的是：

- 工作台已经足够承担管理驾驶舱角色
- 不需要额外做一套 HR 专属高层页面

### 4.10 项目完成

候选人接受 offer 并确认入职后，用户在项目里看到：

- 看板已收敛
- 时间线显示主要阶段完成
- 审批链路完整可追溯
- JD、面试题、反馈模板、offer 审批记录、招聘复盘已归档
- 可复用的面试 rubric、岗位画像模板、offer 沟通话术可沉淀为 Team 资产

此时：

- 项目产物保留在招聘项目中
- 可复用方法论沉淀到 `Team 资产`

这里验证的是：

- 招聘项目完成后，不能只留下一个“已录用”状态
- 仍然需要回放、审计和资产沉淀

## 5. 用当前 Project 框架走查

下面用现有 UI 和项目骨架逐页走查，看这条故事能否走通。

### 5.1 工作台

可以走通。

- HRBP 用 `工作台` 发起项目，查看待审批、最近项目、招聘周报
- 招聘负责人用同一页面切换到更大范围，查看岗位健康、风险热区、成员参与分布
- 不需要新增“HR 驾驶舱”

### 5.2 新建项目

可以走通。

- 创建页已经支持填写目标、范围、约束、可见范围
- 已支持“现在选择 Team”与“稍后挂载 Team”两条路径
- 对招聘项目来说，只需把 Team 模板换成 `招聘 Team`

### 5.3 项目总览

可以走通。

- 能展示岗位阶段、当前阻塞、待审批、最近任务、最近产物
- 请求发起人可以在这里看到审批跟踪
- 项目完成后可以切到完成摘要和回放入口

### 5.4 项目协作台

可以走通。

- 招聘项目天然需要频繁补充要求、更新候选人情况、修改轮次安排
- `协作台` 的核心仍然是承接“项目推进指令”
- `Workstream` 可以自然映射成岗位澄清、候选人筛选、面试协调、offer 审批整改等线程

### 5.5 项目看板

可以走通。

- 看板可以直接承载 JD、搜寻、初筛、面试安排、反馈收集、offer 发放等任务
- `Blocked` 列非常适合表现“待薪酬审批”“待法务确认”“面试反馈未回收”

### 5.6 时间线与风险

可以走通。

- 招聘项目有明确时限，时间线页面很自然
- 风险项可以表达候选人流失、反馈超时、审批滞后、预算不匹配

### 5.7 产物

基本可以走通，但这里有一个边界提醒。

- 产物页可以承载 JD、面试题、反馈模板、offer 建议包、结案复盘
- 受限 artifact 可以承接候选人摘要和薪酬方案
- 但“候选人列表”目前仍更像受限 artifact 或任务上下文，不是显式 ATS 实体

这不是当前框架的阻塞点，但说明：

- 如果后续要深做招聘运营，候选人对象可能要再抽成一等模型
- 在当前限定范围内，先用受限 artifact + 看板任务 + 审批链路就足够

### 5.8 设置

可以走通。

- 项目设置适合管理成员、可见范围、Team 挂载和项目级能力白名单
- 对招聘项目尤其重要的是：谁能看候选人资料，谁只能看阶段摘要

### 5.9 审批中心

可以走通。

- 当前审批中心已经支持“审批人操作”和“发起人只读跟踪”分离
- 薪酬审批、特殊用工条款审批、预算追加都能映射成明确审批请求

### 5.10 资源库 / 知识库 / 定时任务

基本可以走通。

- `资源库` 适合沉淀岗位模板、面试 rubric、招聘复盘
- `知识库` 可以引用面试规范、薪酬政策、用工合规要求
- `定时任务` 可以承接招聘周报、反馈催办、阶段巡检

## 6. 当前框架下的结论

结论是：能走通，但比市场营销场景更接近当前框架边界。

原因不是项目页结构有问题，而是招聘天然更容易逼近两个“尚未一等建模”的对象：

- 候选人对象
- 一对一排期对象

但在当前“先验证 Project 骨架”的目标下，这仍然不是阻塞：

- 候选人先作为受限 artifact 和任务上下文承接
- 排期先作为任务节点、时间线节点和自动提醒承接
- 审批、权限、回放、沉淀都已经有落点

所以当前 `Project` 固定骨架依然成立：

- `总览` 承接全局状态
- `协作台` 承接持续推进
- `看板` 承接任务流转
- `时间线与风险` 承接节奏与异常
- `产物` 承接正式输出与敏感材料
- `设置` 承接权限、挂载和治理

如果后续还要继续压测框架，下一类更值得验证的是“法务合同审查项目”或“采购引入项目”，因为它们会进一步测试：

- 多级审批
- 文档版本对比
- 外部协作方边界
- 更重的合规留痕

但就 HR 招聘场景而言，现有框架已经基本闭环，只是暴露出了未来可能单独建模的对象边界。
