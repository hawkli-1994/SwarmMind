# 市场营销项目用户故事

> 日期：2026-03-27
> 用途：用非研发场景验证 `Project` 框架是否能承接正式业务协作
> 关联文档：`docs/enterprise-crm-user-story.md`、`docs/ui/02-page-map-and-flows.md`、`docs/ui/30-projects-and-project-space.md`

## 1. 故事目标

本文换一个业务场景，不做软件研发，而是做一次正式的市场营销项目，验证以下问题：

- `Project` 是否不仅适用于研发，也适用于市场活动推进
- 当前固定的六个项目子页，是否足以承接营销项目全流程
- `TeamInterfaceAgent` 是否能作为营销 Team 的统一入口
- 审批、资产沉淀、项目回放是否能覆盖内容发布与数据合规场景
- 不新增页面的前提下，现有 `工作台 -> 项目 -> 治理 -> 资产` 结构是否走得通

## 2. 角色

### 2.1 人类角色

- 市场经理：项目发起人，负责提出活动目标、预算和时间要求
- 品牌负责人：审批品牌口径、对外发布素材
- 法务 / 合规负责人：审批活动文案、表单中的用户数据采集字段
- 销售运营同学：提供线索接收规则、反馈线索质量

### 2.2 AI 角色

- `TeamInterfaceAgent`：市场营销 Team 在项目内的统一接口人
- 营销策略 Agent
- 内容策划 Agent
- 视觉设计 Agent
- 渠道投放 Agent
- 数据分析 Agent
- 交付协调 Agent

## 3. 场景概述

一家企业准备在 2026 年第三季度发起一次 B2B 增长活动，为新产品收集销售线索。

市场经理登录 SwarmMind 后，创建一个新的 `Project`，命名为“2026 Q3 线索增长 campaign”。
他在创建页填写：

- 项目目标：6 周内获取 300 个有效 MQL
- 业务范围：活动主题、内容素材、落地页、报名表单、渠道投放、每周复盘
- 约束：预算 50 万、必须走品牌审校、不得采集未审批的个人字段、首期只做 Web 端落地页
- 可见范围：市场部、销售运营、管理层

这个项目创建后，成为活动推进、审批、素材沉淀和复盘的唯一边界。

## 4. 故事主线

### 4.1 创建 Project

市场经理点击“新建项目”后，系统创建：

- `Project`
- 初始 `ProjectMember` / `RoleBinding`
- 空的 `TaskStore`
- 空的 `RunStore`
- 空的项目总览、看板、产物空间

这里验证的是：

- 营销活动也必须先进入正式 `Project`
- 可见性、审计和权限仍然先于执行建立

### 4.2 选择 Team

市场经理在创建页直接选择 `市场营销 Team`，或者先建立空项目，稍后在设置页挂载。

挂载后系统创建：

- `ProjectTeamBinding`
- `ProjectTeamInstance`
- `TeamInterfaceAgent`

此时项目里能看到：

- 已挂载“市场营销 Team”
- 当前状态：待活动澄清
- 接口人 Agent 在线

这里验证的是：

- `Team` 仍然是可复用模板，不是一次性执行实体
- 同一个市场营销 Team 模板可以被多个 campaign 项目复用
- 即使先建空项目、后挂 Team，项目骨架也不会断裂

### 4.3 与接口人 Agent 对话

市场经理打开项目协作台，对 `TeamInterfaceAgent` 说：

“我们要在 6 周内为新产品获取 300 个有效线索。优先渠道是公众号、行业 webinar 和广告投放。需要统一落地页和报名表单，表单字段必须先过法务审批。活动文案不能偏离品牌口径。每周五需要出一份复盘摘要给销售负责人。”

接口人 Agent 做 intake：

- 识别目标、时间、预算和关键约束
- 提炼主要渠道与交付范围
- 标记合规与品牌审批是前置条件
- 建立“Q3 campaign 启动” `Workstream`

然后给出确认摘要：

- 第一阶段目标：启动 Q3 线索增长活动
- 必要交付：活动 brief、内容日历、KV 与落地页、报名表单、渠道计划、周报
- 强约束：品牌审校、法务审批、预算控制、Web 优先
- 关键协作方：市场、品牌、法务、销售运营

用户确认后，Team 正式进入执行。

这里验证的是：

- `TeamInterfaceAgent` 同样适合营销场景，不只适合研发
- 人类输入可以被整理成结构化执行上下文，而不只是聊天记录

### 4.4 工作流模板启动

`市场营销 Team` 自带一套默认工作流模板，例如：

1. 活动目标与 audience 澄清
2. 渠道与内容策略
3. 素材与落地页生产
4. 品牌 / 法务审批
5. 上线投放
6. 周度优化与复盘
7. 项目收尾与资产沉淀

接口人 Agent 确认需求后，系统在项目内自动生成：

- campaign 级任务骨架
- 项目看板
- 时间线与风险摘要
- 初始产物位

用户此时看到的是：

- 当前处于哪一个阶段
- 下一步是谁在推进
- 哪些任务正在进行、阻塞或等待审批
- 哪些正式素材已经产出

这里验证的是：

- 固定 `Project` 骨架并不依赖研发语义
- 看板、时间线、产物对营销项目同样成立

### 4.5 多 Agent 协作展开

随后，营销 Team 内的多个 Agent 进入协作：

#### 营销策略 Agent

- 输出 campaign brief
- 细化目标 audience
- 列出核心渠道与预算建议

#### 内容策划 Agent

- 产出内容日历
- 拆分活动文案、落地页文案、邮件触达文案
- 标出需要品牌统一口径的段落

#### 视觉设计 Agent

- 产出主视觉方向稿
- 产出落地页线框和广告素材草图

#### 渠道投放 Agent

- 拆分渠道投放计划
- 标记启动前依赖，例如素材齐备、表单审批通过

#### 数据分析 Agent

- 设计周报指标
- 定义线索质量、转化率、渠道成本等观测项

用户在项目里持续看到：

- 看板卡片变化
- 时间线推进
- 任务 handoff
- 文案、视觉、投放计划、周报模板等 artifact 逐步形成

这里验证的是：

- `RunStore`、`ArtifactStore`、`AuditLog` 对营销项目依然是一等数据
- 项目控制面关注的是推进过程，而不只是最后一份总结

### 4.6 用户中途干预

活动推进到第二周，市场经理提出新要求：

- “广告预算先从 20 万下调到 12 万，把重点放到 webinar 报名。”
- “报名表单先不要采手机号，只保留公司、姓名、邮箱和职位。”
- “落地页标题需要更偏行业痛点，不要写得像品牌宣传。”

这些意见进入 `TeamInterfaceAgent` 后，被转换为：

- 预算调整任务
- 文案改写任务
- 表单字段变更
- 风险与时间线更新

这里验证的是：

- 人类可以在推进中持续校正方向
- `协作台` 能承接范围变更，不要求用户跳出项目再重建流程

### 4.7 审批暂停与恢复

在“表单上线”前，系统识别到两类审批：

- 对外发布文案需要品牌负责人审批
- 报名表单采集字段需要法务 / 合规负责人审批

此时项目状态切换为 `等待审批`，系统生成明确的审批请求，包含：

- 请求类型：品牌审校 / 数据采集合规审批
- 关联项目与当前阶段
- 关联 artifact：落地页文案、报名表单字段说明
- 风险等级与影响范围
- 若不批准，将阻塞哪些任务

市场经理在项目总览里看到：

- 当前阻塞点：品牌审校未完成、表单字段待法务确认
- 当前状态：等待审批
- 已提交给谁：品牌负责人、法务负责人

但他本人没有审批权，所以只能跟踪，不能自行批准。

审批通过后：

- 结果进入 `AuditLog`
- 项目自动恢复推进
- 渠道投放与落地页发布任务重新进入执行

这里验证的是：

- 当前审批中心模型并不依赖研发特定场景
- 对外发布和数据采集也能被统一抽象成明确审批对象

### 4.8 管理者在工作台巡视

市场负责人登录后，在自己的 `工作台` 中看到：

- 市场部当前进行中的项目
- 哪些项目在等待审批
- 哪些项目风险高
- 哪些成员正在参与多个项目
- 哪个 Team 当前最忙

点击“2026 Q3 线索增长 campaign”后，他进入项目总览，再下钻到看板、时间线、产物和审批跟踪。

这里验证的是：

- 工作台已经足够承担“管理驾驶舱”角色
- 不需要为市场负责人再单独造一套项目外页面

### 4.9 项目完成

活动结束后，用户在项目里看到：

- 看板已收敛
- 时间线显示主要阶段完成
- 最终线索结果、渠道表现和预算消耗已归档
- 审批链路完整可追溯
- 复盘结论与高表现素材可沉淀为 Team 资产

此时：

- campaign brief、素材包、周报、结案复盘留在项目产物里
- 可复用的模板、优胜文案、复盘方法论沉淀到 `Team 资产`

这里验证的是：

- 营销项目的“结果”不只是文件，也包括回放、决策和可复用资产
- `Project` 仍然是执行与归档边界

## 5. 用当前 Project 框架走查

下面用现有 UI 和项目骨架逐页走查，看这条故事能否走通。

### 5.1 工作台

可以走通。

- 市场经理用 `工作台` 发起项目，查看待审批、最近项目、今日推进摘要
- 市场负责人用同一页面切换到更大范围，查看项目健康、风险热区、成员参与分布
- 不需要新增“市场驾驶舱”

### 5.2 新建项目

可以走通。

- 创建页已经支持填写目标、范围、约束、可见范围
- 已支持“现在选择 Team”与“稍后挂载 Team”两条路径
- 对营销项目来说，只需把 Team 模板选项换成 `市场营销 Team`

### 5.3 项目总览

可以走通。

- 能展示项目阶段、阻塞、待审批、最近任务、最近产物
- 请求发起人可以在这里看到审批跟踪
- 项目完成后可以切到完成摘要和回放入口

### 5.4 项目协作台

可以走通。

- 营销项目也需要持续 intake、补充意见、修改口径、调整渠道与预算
- `协作台` 的核心不是写代码，而是承接“项目推进指令”
- `Workstream` 可以自然映射成 campaign 启动、素材生产、审批整改、投放优化等线程

### 5.5 项目看板

可以走通。

- 看板卡片可直接承载 brief、文案、素材、落地页、表单、投放、周报等任务
- `Blocked` 列非常适合表现“待品牌审批”“待法务审批”“素材未齐”

### 5.6 时间线与风险

可以走通。

- 市场活动天然有强时间边界，时间线页面对营销项目是刚需，不是附属页
- 风险项可以表达审批延迟、渠道成本超预算、报名转化不达预期、销售跟进资源不足

### 5.7 产物

可以走通，但有一处轻微语义偏研发。

- 产物页完全能承载 brief、文案、视觉稿、落地页线框、周报、结案复盘
- 沉淀为 `Team 资产` 的动作也适合营销项目
- 目前线框示例里有 `代码` 类型字样，后续只需把 artifact 类型做成可配置标签，不需要增页

### 5.8 设置

可以走通。

- 项目设置适合管理成员、Team 挂载、可见范围和项目级能力白名单
- 若营销项目后续需要接广告平台、表单系统、CRM 等能力，也仍然属于现有设置页范围

### 5.9 审批中心

可以走通。

- 当前审批中心已经支持“审批人操作”和“发起人只读跟踪”分离
- 品牌审校、法务合规、预算追加都能映射成明确审批请求

### 5.10 资源库 / 知识库 / 定时任务

基本可以走通。

- `资源库` 适合沉淀 campaign 模板、优秀素材、结案复盘
- `知识库` 可以作为品牌规范、行业资料、FAQ 的引用来源
- `定时任务` 可以承接每周复盘摘要、固定报表刷新等自动动作

## 6. 当前框架下的结论

结论是：能走通，而且不需要扩张项目框架。

真正需要变化的不是页面结构，而是以下三类“配置层”内容：

- Team 模板：从 `软件开发 Team` 扩展到 `市场营销 Team`
- 工作流模板：把研发阶段替换为营销阶段
- artifact 与审批类型：从“PRD / 代码 / 技术权限”扩展到“brief / 素材 / 品牌审校 / 合规审批”

也就是说，当前 `Project` 固定骨架是成立的：

- `总览` 承接全局状态
- `协作台` 承接持续推进
- `看板` 承接任务流转
- `时间线与风险` 承接节奏与异常
- `产物` 承接正式输出与沉淀
- `设置` 承接治理与挂载

如果后续要继续验证另一类业务，下一篇更值得写的会是 `HR 招聘项目用户故事`。因为它会进一步检验：

- 多轮审批
- 候选人资料权限
- 面试排期与定时任务
- 项目完成后的知识沉淀

但就市场营销场景而言，现有框架已经足够闭环。
